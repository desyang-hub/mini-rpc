## 协议层设计

### 第 4-7 天：解决“粘包/拆包”问题 (面试必问)
* 痛点：TCP 是流式协议，发送 "ABC" 和 "DEF"，接收端可能收到 "ABCDEF" 或者 "AB"、"CDEF"。
* 任务：设计应用层协议。
* 方案：长度前缀法（最常用）。在消息头加 4 个字节表示消息体长度。
* 产出：一个能正确收发完整数据包的 TcpServer 和 TcpClient。


### protocol

`[4byte][body]` 这种格式是解决 TCP 粘包问题的**基础且正确**的思路，也就是“长度前缀法”。它通过一个4字节的整数来声明后续 `body` 的长度，让接收方能够明确消息的边界。

然而，对于一个健壮、可维护的 RPC 框架来说，仅有长度信息是**远远不够的**。这只是一个起点，直接使用这种极简格式会面临诸多问题。

一个成熟的 RPC 协议通常会在长度字段的基础上，增加更多的元数据来构成一个完整的消息头（Header），以支持更复杂的功能和保证通信的可靠性。

### 🧐 为什么仅有 `[4byte][body]` 是不够的？

1.  **缺乏协议识别能力 (Magic Number)**
    *   **问题**：接收方收到数据后，无法判断这串数据是否符合预期的协议。如果端口被错误连接或受到恶意攻击，接收方会尝试将一个随机数据流的开头4字节解析为长度，这可能导致程序崩溃或产生不可预知的错误。
    *   **解决方案**：在协议开头增加一个固定的“魔数”（Magic Number），例如 `0x5250` ('R''P')。接收方首先校验魔数，如果不匹配则直接丢弃或报错，实现快速失败。

2.  **缺少版本控制 (Version)**
    *   **问题**：当 RPC 协议需要升级时（例如增加新的字段或改变语义），旧版本的客户端和服务端可能无法兼容，导致调用失败。
    *   **解决方案**：增加一个版本号字段。服务端可以根据版本号来兼容不同版本的客户端，实现平滑升级。

3.  **无法区分消息类型 (Message Type)**
    *   **问题**：RPC 通信中不只有请求（Request）和响应（Response），通常还需要心跳包（Heartbeat）来维持长连接、探测对方是否存活。仅有长度信息，接收方无法区分收到的数据是业务请求还是心跳包。
    *   **解决方案**：增加一个消息类型字段，明确标识当前消息是请求、响应还是心跳包。

4.  **序列化与压缩方式不明确 (Serialization & Compression ID)**
    *   **问题**：`body` 是一串二进制数据，接收方需要知道它是如何被序列化的（如 Protobuf, JSON, Hessian 等）以及是否被压缩（如 Gzip, Snappy），才能正确地反序列化和解压。
    *   **解决方案**：在消息头中增加序列化ID和压缩算法ID，让接收方能够根据这些ID选择正确的解码器和解压器。

5.  **缺少请求与响应的关联 (Request ID)**
    *   **问题**：在异步调用或并发调用场景下，客户端会发出多个请求。当响应返回时，客户端需要知道这个响应是对应哪一个请求的。
    *   **解决方案**：客户端为每个请求生成一个唯一的 `request_id`，并在请求头中发送。服务端在响应中原样返回这个 `request_id`，客户端据此匹配响应和请求。

6.  **缺乏数据完整性校验 (Checksum)**
    *   **问题**：数据在网络传输过程中可能因网络波动等原因发生损坏。仅靠 TCP 的校验和不足以保证应用层数据的完整性。
    *   **解决方案**：增加一个校验和字段（如 CRC32），对 `body` 进行计算。接收方收到数据后重新计算并比对，如果不一致则说明数据已损坏，可以要求重传。

### 📦 一个更健壮的 RPC 协议头设计示例

一个工业级的 RPC 协议头会包含上述多种信息。以下是一个简化的示例，展示了如何从 `[4byte][body]` 演进到一个更完整的结构：

| 字段名 | 大小 | 说明 |
| :--- | :--- | :--- |
| **Magic** | 2 字节 | 魔数，用于快速识别协议 |
| **Version** | 1 字节 | 协议版本号 |
| **Message Type** | 1 字节 | 消息类型（请求/响应/心跳） |
| **Serialization ID** | 1 字节 | 序列化方式（Protobuf/JSON等） |
| **Compression ID** | 1 字节 | 压缩算法（Gzip/Snappy等） |
| **Request ID** | 8 字节 | 请求唯一标识，用于匹配响应 |
| **Body Length** | 4 字节 | 消息体的长度 |
| **Checksum** | 4 字节 | 消息体的 CRC32 校验和 |
| **Reserved** | 4 字节 | 预留字段，用于未来扩展 |
| **Body** | 变长 | 实际的消息数据 |

综上所述，`[4byte][body]` 是解决 TCP 消息边界问题的正确方向，但一个生产级别的 RPC 协议需要在此基础上设计一个更丰富的消息头，以支持协议识别、版本兼容、功能扩展和数据安全。
